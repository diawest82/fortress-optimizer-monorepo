## 🎯 How It Answers Your Question

**Your Question:** "When user signs up either in the tool, or via the website, how is their access granted, maintained, and tier known?"

**Complete Answer:**

```
1. USER SIGNS UP
   POST /auth/signup {email, password}
   ↓
2. ACCOUNT CREATED IN DATABASE
   users_db: {id, email, password_hash, created_at}
   subscriptions_db: {user_id, tier: "free", status: "active"}
   ↓
3. ACCESS GRANTED
   Returns JWT token (24-hour session)
   Returns API key (sk_xxx for API access)
   ↓
4. TIER MAINTAINED
   subscriptions_db tracks user's current tier
   Updated when user upgrades via /billing/upgrade
   Enforced on every request via verify_api_key_with_tier()
   ↓
5. TIER KNOWN
   Every request includes user tier in context
   Tier determines:
   - Rate limit (10/100/1000 req/min)
   - Monthly quota (50K/500K/50M tokens)
   - Feature access (all_providers flag)
   ↓
6. ACCESS CONTROLLED
   POST /optimize:
   ├─ Check rate limit → 429 if exceeded
   ├─ Check monthly quota → 429 if exceeded
   ├─ Check provider access → 403 if restricted
   └─ Execute optimization if all checks pass
   ↓
7. USER UPGRADES (Optional)
   POST /billing/upgrade {tier: "pro"}
   subscriptions_db updated: tier = "pro"
   ↓
8. NEXT REQUEST SEES NEW TIER
   API key validated → user_id retrieved
   User tier looked up → "pro" (100 req/min, all providers)
   Old limits no longer enforced
```

## 🔐 Security Measures

✅ **Password Security**
- PBKDF2-HMAC-SHA256 with salt
- 100,000 iterations (NIST recommendation)
- Per-user random salt
- Constant-time verification

✅ **API Key Security**
- Generated with cryptographic randomness
- Hashed (SHA256) before storage
- Never logged or displayed after creation
- Can be rotated by creating new keys

✅ **JWT Token Security**
- Signed with HS256 algorithm
- 24-hour expiration
- Includes user_id, email, tier
- Refresh endpoint for token rotation

✅ **Rate Limiting**
- Per-user tracking (not global)
- 60-second rolling window
- Tier-specific limits enforced
- Returns 429 when exceeded

✅ **Quota Enforcement**
- Monthly usage tracked per user
- Reset on 1st of each month
- Enforced before each optimization
- Returns 429 when exceeded

✅ **Feature Access Control**
- Tier-based provider restrictions
- Free tier: OpenAI only
- Pro/Team: All 16 providers
- Returns 403 for restricted features

## 📊 Database Schema

### In-Memory (Development)
```python
users_db: {user_id → {id, email, password_hash, created_at}}
api_keys_db: {key_hash → {user_id, name, created_at, expires_at}}
subscriptions_db: {user_id → {tier, status, stripe_id, dates}}
usage_db: {"user_id:YYYY-MM" → {tokens, optimizations}}
rate_limit_store: {user_id:rate_limit → [request_timestamps]}
```

### PostgreSQL (Production)
- 5 tables: users, api_keys, subscriptions, usage, rate_limit
- Foreign keys linking user_id across tables
- Indexes on email, api_key_hash, user_id
- Timestamp triggers for created_at/updated_at

```bash
  -H "Authorization: Bearer sk_xxx"

# Optimize tokens
curl -X POST http://localhost:8000/optimize \
  -H "Authorization: Bearer sk_xxx" \
  -H "Content-Type: application/json" \

# Returns: {status, saved_tokens, remaining_monthly_tokens}

# Upgrade to Pro
curl -X POST http://localhost:8000/billing/upgrade \
  -H "Authorization: Bearer jwt_token" \
  -H "Content-Type: application/json" \
  -d '{"tier":"pro","billing_cycle":"monthly"}'
# Returns: {tier: "pro", price: 9.99, ...}
```
